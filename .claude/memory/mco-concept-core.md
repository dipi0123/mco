2026-07-09 브레인스톰에서 정렬된 컨셉 코어 (현재 합의):

- **MCO (Memory Context Orchestrator)** = 사용자의 메모리를 관리하고, 프롬프트를 해석해 **계정·프로젝트·에이전트 단위**로 필요한 컨텍스트만 추출해 LLM/에이전트에 제공하는 **오케스트레이션 레이어**. GPT·Claude를 대체하는 게 아니라 그 **앞단에서 개인의 메모리를 다뤄주는 '메모리 OS' 계층**.
- **왜 레이어인가:** 각 LLM은 자체 메모리를 갖지만 **플랫폼 밖으로 이어지지 않는다**. MCO는 **MCP 서버 + OAuth 기반 인증(계정 바인딩)** 으로 사용자 소유의 메모리를 불러와, 어떤 LLM을 쓰더라도 일관된 메모리 컨텍스트를 제공한다. 대상은 빅테크 LLM만이 아니라 **사용자 정의 에이전트**까지.
- **스몰 메모리 모델 = 코어.** MCO 안의 작은 모델이 담당: ① 기억할 가치 판단 ② 저장 범위 결정 ③ 충돌 감지 ④ 갱신 ⑤ 필요한 기억 선택. 스스로 약간의 학습(개인화)도 한다. **단순 메모리 RAG 아님** — 개인화 히스토리 관리 + RAG + 자가 학습이 결합된 형태.
- **경계:** 세션(단기) 기억은 기존 LLM이 충분히 처리 → MCO는 **세션 밖의 지속성 있는 메모리 관리에 집중**.

**Why:** 이 정의가 모든 산출물(차터·스펙·피치)의 척추다. '메모리 저장소'나 '메모리 RAG'로 축소되는 순간 컨셉이 죽는다.

**How to apply:** 산출물에서 MCO를 항상 '레이어/오케스트레이터'로 그리고, 스몰 모델 5역할을 전면에 유지한다. [[mco-selection-over-storage]] · [[mco-positioning-bigtech-gap]] · [[mco-memory-structure]].
