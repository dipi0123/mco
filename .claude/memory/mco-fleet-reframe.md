2026-07-10 정렬·적용된 리프레임 (현재 합의, 차터 v0.2 반영). 계기: OpenAI 에이전트 루프 발표(Manager of Managers·inner/outer loop·주의력 병목) 분석 + 경쟁 지형 리서치.

- **디자인 센터 이동**: '여러 LLM을 오가는 사람'(1.0) → '에이전트 함대를 지휘하는 사람(Manager of Managers)'(2.0). 1.0은 폐기가 아니라 **함대 크기 1의 특수해**로 흡수 — 크로스 플랫폼 ⊂ 크로스 런타임.
- **소비자→생산자**: 에이전트 런의 학습(실패한 접근 포함)이 가치 판단 게이트(역할 ①)를 거쳐 지속 메모리로 환류(write-back). 구조적 페인: 비동기·트리거 구동 에이전트 세계에선 사람이 함대의 '메모리 버스'를 못 한다.
- **5역할의 함대 직무**: ① 런 배기가스 트리아지 ③ 에이전트 간 충돌 → 사람의 결정 큐 ④ TTL·출처 관리(백로그→코어 승격) ⑤ 위임 시 브리핑 팩 생성.
- **MVP 웨지 스왑**: 하네스 플러그인(Claude Code·Codex) + MCP 서버 + 결정 인박스형 대시보드. 브라우저 확장은 phase 2. 타깃 = 코딩 에이전트 파워유저 먼저. **(→ 2026-07-12 v0.3 재개정: 웹 커넥터 우선·프로슈머 페르소나 — [[mco-wedge-web-connector]].)**
- **가드레일**: 에이전트 스케줄링·라우팅 안 함(오케스트레이션 프레임워크 아님 — 그건 하네스 몫), 관측/트레이싱 제품 아님(배기가스는 입력, 제품은 기억).
- **최우선 검증**: write-back 정밀도 — 실제 에이전트 세션 로그에 역할 ① 트리아지를 돌려 precision 수동 채점(코드 이전 가능). 이게 안 되면 2.0 전체가 무너진다.

**Why:** 2026-07 경쟁 리서치가 1.0 단독(이동성)은 혼잡 시장, 2.0 프리미티브 조합(스코프·게이트·브리핑 팩·정밀 선택)은 실측 공백임을 확인 — 디자인 센터 결정이 취향이 아니라 증거 문제가 됨.

**How to apply:** 모든 산출물에서 함대 서사를 기본으로, 채팅 유스케이스는 특수해로 서술. [[mco-concept-core]] · [[mco-competitive-landscape]] · [[mco-mvp-direction]]
