2026-07-11 세션 판정 (근거 정본: `docs/리서치/MCO_사업성분석_v0_1.md` §2):

- **실체**: Nous Research, 2026-02-25 출시, ★212K/4.5개월, 릴리즈 ~2주. 기억 = MEMORY.md+USER.md **1,300토큰 하드캡**(동결 스냅샷) + SQLite FTS5 + 외부 메모리 프로바이더 슬롯 9개(mem0·Supermemory·Honcho 등). 스코프 벽·TTL·출처·충돌 해소 **없음**. "md 기반 히스토리 관리" 특성화는 부분적으로만 맞음.
- **판정 — 경쟁자 아닌 3역**: ① 수요 증명("나를 기억하는 AI"가 대중 스케일) ② 런타임 커모디티화(지속 가치는 기억으로 이동) ③ 유통 채널(프로바이더 슬롯 = 자기선별된 ICP 저수지). **정면 경쟁(호스티드 개인 에이전트) 금지** — 속도·무료·규모에서 자살 코스.
- **포지셔닝 문장**: *Hermes = 기억을 가진 에이전트 하나. MCO = 당신의 모든 에이전트가 공유하는 기억 하나.* 컨셉·약속은 유사, 구조는 역(에이전트 內 기능 vs 에이전트 횡단 계정).
- **Hermes Cloud (2026-07-07~08)**: 호스티드+팀+통합과금, "Memory lives with the agent, not the device" — 멀티디바이스를 에이전트 호스팅으로 해결. 기억의 독립 계정화는 로드맵에 없음(셀프호스트 동기화 P3 방치).
- **차용**: 상한 강제 큐레이션 / 메모리 도구 인터페이스(초과 시 에러, 침묵 삭제 금지) / 주입 전 보안 스캔. **개선**: 전역→스코프 벽, 동결 스냅샷→서버사이드 라이브 갱신.
- **슬롯 실측 공백**: 프로바이더 회상 노이즈 — Supermemory 19/36·mem0 20/36 낡음/엉뚱(독립 벤치). 내장은 노이즈 적으나(2/36) 회수 실패. → 슬롯 승부처 = 정밀도.

Why: "Hermes 쓰면 되는데 왜 MCO?"의 답이 페르소나로 갈림 — 컨텍스트 1개인 유저는 Hermes로 충분(우리 고객 아님), 멀티 컨텍스트(법인·클라이언트·표면·에이전트 N개) 유저는 에이전트-바운드 기억으로는 구조적으로 불가.

How to apply: 데모 목표 = "원 브레인, 멀티 서피스"(같은 질문을 Claude 웹·Claude Code·Hermes+MCO에서 → 같은 브리핑·같은 결론). 킬 트리거: Nous가 메모리를 Tool Gateway식 자체 번들 시 슬롯 전략 재검토. [[mco-viability-verdict]] [[mco-competitive-landscape]]
