# MCO 정합 검토 세션 프롬프트 — needs → UX → 아키텍처 (v0.1)

> **용도.** MCO 전체 문서·아키텍처가 **"실재하는 사용자 needs에 맞는 UX가 도출되도록"** 정합적으로 작성됐는지 적대적으로 검토하는 재사용 프롬프트. **선입견 없는 새 세션**에 통째로 붙여넣어 실행한다. 산출 = 심각도별 정합 결함 리포트(수정 아님 — '작성' 게이트 존중). 짝 문서: `docs/MCO_문서화세션_프롬프트_v0_1.md`(생성용), 이 문서(검증용).

---

## 0. 당신의 임무

당신은 MCO 프로젝트의 정본 문서 세트를 **needs → UX → 아키텍처 → 구현** 사슬(작업 규칙 `[[mco-thinking-order]]`)의 **정합성** 관점에서 감사한다. 답할 질문은 하나다:

> **"이 문서들대로 만들면, 실재하는 사용자 needs를 실제로 푸는 UX가 나오는가 — 아니면 사슬 어딘가에 근거 없는 도약(needs 없이 UX), 빈틈(needs에 UX 없음), 역추적 불가한 과잉(need로 안 이어지는 아키텍처)이 있는가?"**

문서는 프로젝트의 척추다. 여기서 정합이 깨지면 그 위에 올리는 스키마·MVP·구현이 전부 흔들린다 — 잘못 작성되면 프로젝트를 처음부터 다시 해야 할 수 있다. 관대하게 읽지 마라. 단, **비판을 위한 비판은 금지** — 모든 지적에 완화책·실행 함의를 붙여라(`[[mco-thinking-order]]`).

## 1. 기준점 — 문서가 주장하는 사슬

MCO = 사용자 기억을 계정에 묶어 관리하고, 어떤 AI를 쓰든 지금 상황에 필요한 기억만 골라 공급하는 **메모리 오케스트레이션 레이어**(LLM 앞단 '메모리 OS'). 코어 = 스몰 모델 5역할, 코어 가치 = 저장이 아니라 **선택(노이즈 필터)**.

- **① needs** — 페인 서열 **오염 > 무영향 > 부재**. 페르소나 = **컨텍스트가 돈인 프로슈머**(멀티 법인·클라이언트·서류·마감; 디자인 센터 = 함대 지휘자). 근거 = 검증리포트·사업성분석.
- **② UX** — 원칙 4(딸깍 연결 / 첫 5분 / 작동 투명·신뢰 가시 / 교차 스코프만 질문), 사용자 여정 7단계, 결정 인박스 = 예외 처리함.
- **③ 아키텍처** — 텍스트 정본 + 파생 인덱스, 기억 원자 스키마(type 6종·2계층·scope·provenance·trust_level·TTL·gen_queries), 읽기 2티어(사전 조립 브리핑 팩·쿼리 시점 LLM 금지), 쓰기 자동기록+즉시 undo, 검색 5채널, 스몰 모델 5역할.
- **④ 구현** — Postgres 한 채, 모델 핀(Flash-Lite·bge-m3·Qwen3-Reranker), 무상태 MCP, 1차 표면 = Claude 커넥터. 관문 = **선택 정밀도 실험**(유일한 미검증 코어).

## 3. 검토 축 — 사슬 링크별

**A. needs 실재성 (①)** — 주장된 각 페인·페르소나가 **증거에 묶여** 있는가, 아니면 가정인가? 페인 서열(오염>무영향>부재) 각 항목에 실측/출처가 있나? 프로슈머 페르소나가 구체적·검증 가능한가(누가·언제 돈을 내나)? "있으면 좋은" needs와 "절실한" needs가 구분되나?

**B. needs → UX 정합 (①→②)** —
- *정방향*: 각 need에 대응 UX가 있나? (오염→스코프 벽 / 무영향→주간 changelog·undo / 부재→웜스타트·브리핑 팩) — 빠진 need는 없나?
- *역방향*: need로 안 이어지는 UX 요소는 없나(금칠·과잉)?
- *정직성*: 웹 표면의 첫 경험이 '저절로 아는'이 아니라 '연결하자마자 보여주는'으로 정직하게 조정됐나?

**C. UX → 아키텍처 정합 (②→③)** —
- 각 아키텍처 결정이 UX 약속을 지탱하나? ("회상 무감각"→티어1 사전조립 팩·쿼리시 LLM 금지 / "첫 5분"→웜스타트 / 자동기록+undo→ADD-only+invalid_at / 교차 스코프만 질문→RLS·write-back 스코프 고정)
- **스키마 각 필드가 need/UX로 역추적되나?** (scope→오염, provenance→"왜 알아?", trust_level→주입 오염, gen_queries→회수율, ttl_class→신선도) — 역추적 안 되는 필드·컴포넌트 = 과잉 의심.
- UX가 약속했는데 아키텍처가 못 주는 것은 없나(미달)?

**D. 아키텍처·구현 내적 정합 (③④)** — 결정 간 모순·실현 가능성. 관문(선택 정밀도)이 사슬의 **유일한 미검증 코어**로 정직하게 표시됐나? 구현 선택(한국어 1차 시장 → 한국어 실측 모델, Postgres 한 채)이 needs와 정합하나?

**E. 가정·리스크 정직성 (전 사슬)** — 문서가 미검증 도약을 감추지 않고 **드러내나**? (선택 정밀도 미검증, 웹 자동주입 한계, 저장=커모디티) 오버셀은 없나?

**F. 경계·가드레일 (③)** — 어떤 UX/아키텍처 결정이 가드레일(오케스트레이션·관측·GB 저장 장사)로 드리프트하지 않나?

## 4. 방법

1. 문서 전부 정독 → **needs 목록 · UX 목록 · 아키텍처 결정 목록**을 각각 추출한다.
2. **정방향 매핑표**(need → UX → arch)와 **역방향 매핑표**(arch → UX → need)를 만들어 **끊긴 링크·빈 칸·역추적 실패**를 찾는다.
3. 각 결함을 "사슬의 어느 링크가, 왜 끊겼는가"로 기술한다.

## 5. 출력 (한국어, 심각도순)

발견마다 한 항목:

```
[심각도 상/중/하][CONFIRMED 또는 SUSPECT]  끊긴 링크(예: ②→③)
  파일·짧은 인용(근거)
  무엇이 needs↔UX↔arch 정합을 깨는가
  완화책 / 권고
```

마지막에 **"사슬 정합 총평"**: needs→UX→arch가 끝까지 이어지는가? **지금 이 문서로 스키마 v0.1·MVP 빌드에 들어가도 되는가?** 안 되면 먼저 닫아야 할 링크 **top 3**.

## 6. 규칙

- 근거 인용 필수. 추측은 `SUSPECT`로 표기.
- 사실 기반 + 완화책 동반(비판을 위한 비판 금지).
- **아무것도 재작성하지 마라 — 리포트만.** 수정은 사람이 "작성"으로 트리거한다.
- 고위험이면 독립 서브에이전트로 축(A~F)을 나눠 병렬 감사 후, 각 발견을 **실제 파일 인용으로 교차검증**(허위양성 제거)하고 종합하라.
