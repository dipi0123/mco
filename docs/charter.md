# MCO (Memory Context Orchestrator) — 프로젝트 지침 (Project Charter) v0.1

> 이 파일이 차터(지침) **정본**이다 — repo 통합 베이스. 2026-07-09 브레인스톰 정리본 기반 **v0.1 draft**. 브레인스토밍 단계이므로 모든 조항은 '잠김'이 아닌 **현재 합의**이며, 논의로 갱신한다(갱신 시 `.claude/memory/` 동기화).

---

## 1. 시스템 정의

**MCO는 사용자의 메모리를 관리하고, 프롬프트를 해석해 계정·프로젝트·에이전트 단위로 필요한 컨텍스트만 추출해 LLM이나 에이전트에 제공하는 오케스트레이션 레이어다.** GPT·Claude 등 LLM을 대체하지 않는다 — 그 **앞단에서 개인의 메모리를 다뤄주는 '메모리 OS' 계층**이다.

각 LLM은 자체 메모리를 갖고 있지만 **플랫폼 밖으로 이어지지 않는다**. MCO는 **MCP 서버와 OAuth 기반 인증**을 통해 사용자 소유의 메모리를 불러오고, **어떤 LLM을 쓰더라도 일관된 메모리 컨텍스트**를 제공한다. 대상은 빅테크 LLM만이 아니라 **사용자 정의 에이전트**까지 포함한다.

## 2. 문제 · 포지셔닝 — 빅테크 빈틈

개인화 메모리 자체는 차별점이 아니다 — 빅테크 모델들이 이미 각자 관리한다. MCO의 공략점은 **"빅테크가 하기엔 짜치고(세밀·귀찮), 사용자는 절실한" 빈틈**이다:

- **크로스 플랫폼 이동성** — GPT에서 쌓인 나의 정보를 Claude에서도, Claude에서 쌓인 것을 GPT에서도 쓴다. 개념 앵커 = **Personal Memory Passport**: 플랫폼·계정이 달라도 내 메모리를 권한에 맞게 동기화해 필요한 것만 제공.
- **계정 바인딩 · 사용자 소유** — 기억은 벤더가 아니라 **사용자 계정에 바인딩**된다. OAuth 소스 인증으로 어디서든 서비스받는다.
- **에이전트별 기억 권한 스코프** — 에이전트마다 필요한 도메인의 기억만 부여한다. 빅테크 입장에선 너무 세밀하고 귀찮지만, 사용자에겐 절실한 영역.

## 3. 코어 — 스몰 메모리 모델

MCO 안의 **작은 모델**이 메모리 지능을 담당한다. 단순 메모리 RAG가 아니다 — **개인화 히스토리 관리 + RAG + 약간의 자가 학습**이 결합된 형태다.

| # | 역할 | 내용 |
|---|---|---|
| ① | 기억 가치 판단 | 입력 중 지속 보관할 가치가 있는 정보 식별 |
| ② | 저장 범위 결정 | 어느 스코프(계정·프로젝트·에이전트)에 저장할지 결정 |
| ③ | 충돌 감지 | 기존 기억과 모순되는 새 정보 탐지 |
| ④ | 갱신 | 낡은 기억을 새 사실로 교체·무효화 |
| ⑤ | **기억 선택** ★ | 지금 컨텍스트에 필요한 기억만 인출 — 코어 |

## 4. 메모리 구조 (구상)

- **수집** — 입력 데이터 중 지속적으로 쓸 만한 기억(나에 대한 정보)을 리스트업한다.
- **구조화 (3층)** — ① 명시적으로 저장된 정보 ② 맥락에서 추출한 의미 ③ 사건의 조각들(에피소드).
- **선택** — 질문이 들어오면 **필요한 기억만** 꺼낸다. 무관한 기억을 걸러내는 **노이즈 필터**가 코어 가치다(실패 기준: 애플워치 논의에 Setto 기억 주입). 평가 축 = recall보다 **precision 우선**.

## 5. 경계 (범위)

- **세션(단기) 기억은 기존 LLM이 충분히 처리한다.** MCO는 **세션 밖의 지속성 있는 메모리 관리**에 집중한다.
- MCO는 LLM 추론 자체를 하지 않는다 — 컨텍스트를 준비해 **제공**하는 레이어다.

## 6. 제공 형태

- **MCP 서버** — 어떤 LLM/에이전트든 MCP 연동으로 MCO 메모리를 사용한다.
- **OAuth 계정 인증** — 기억 = 사용자 계정 소유. 소스 인증으로 서비스한다.
- **권한 스코프** — 계정·프로젝트·에이전트 단위로 컨텍스트를 분리 제공한다.

## 7. MVP 구상

- 구성: **MCP 메모리 서버 + 브라우저 확장 + 대시보드**.
- 초기 타깃: **파워유저** — 개발자·컨설턴트·변호사 등 여러 LLM/에이전트를 오가며 맥락 단절 페인이 절실한 직군.

## 8. 버티컬 — 제조

제조업 적용 시 **'LLM 독립형 메모리 레이어' 표현을 쓰지 않는다.** 정본 포지셔닝:

> **"현장의 설비·품질·작업자 이력을 기억하고, 지금 상황에 딱 맞는 과거 사례를 꺼내주는 제조 메모리 컨텍스트"** (Factory Memory Context)

'지금 상황에 맞는 과거 사례 인출' 라인으로 시작해야 설득력이 높다.

## 9. 미결 (백로그)

논의로 확정할 것들 — 확정 시 본 차터 갱신 + `.claude/memory/` 기록:

- **스몰 모델 실체** — 어떤 모델을 어떻게(파인튜닝·호스팅·온디바이스), '약간의 자가 학습'의 구현 방식.
- **메모리 스키마** — 3층 구조의 저장 스키마·충돌 해소 규칙·스코프 모델.
- **프라이버시·보안 모델** — 기억 = 최고 민감 데이터. 암호화·삭제권·감사 로그.
- **수집 경로** — 브라우저 확장 외 수집 소스(MCP 양방향·API 등).
- **경쟁·시장 리서치** — 기존 메모리 레이어(mem0·Zep·Letta류)와 빅테크 각사 메모리 기능 현황.
- **수익 모델 · 서비스 네이밍 · 제조 버티컬 구체 시나리오.**

## 10. 용어

| 용어 | 정의 |
|---|---|
| **MCO** | Memory Context Orchestrator — 본 프로젝트 정본 명칭 |
| **Personal Memory Passport** | 크로스 플랫폼 메모리 이동성의 개념 앵커 |
| **스몰 메모리 모델** | MCO 내부에서 5역할을 수행하는 작은 모델 |
| **기억 선택** | 상황에 필요한 기억만 인출하는 코어 기능(역할 ⑤) |
| **제조 메모리 컨텍스트** | 제조 버티컬 정본 포지셔닝 (Factory Memory Context) |
