## 1. 시스템 정의

**MCO는 사용자의 메모리를 관리하고, 프롬프트를 해석해 계정·프로젝트·에이전트 단위로 필요한 컨텍스트만 추출해 LLM이나 에이전트에 제공하는 오케스트레이션 레이어다.** GPT·Claude 등 LLM을 대체하지 않는다 — 그 **앞단에서 개인의 메모리를 다뤄주는 '메모리 OS' 계층**이다.

각 LLM은 자체 메모리를 갖고 있지만 **플랫폼 밖으로 이어지지 않는다**. MCO는 **MCP 서버와 OAuth 기반 인증**을 통해 사용자 소유의 메모리를 불러오고, **어떤 LLM/에이전트를 쓰더라도 일관된 메모리 컨텍스트**를 제공한다.

**v0.2 — 에이전트는 기억의 소비자이자 생산자다.** 에이전트가 런(run) 중 획득한 학습(성공 경로·실패한 접근·제약 발견)은 **가치 판단 게이트(역할 ①)를 거쳐 지속 메모리로 환류(write-back)** 된다. 사람의 채팅 세션은 '함대 크기 1'의 특수해다 — 크로스 플랫폼 메모리(1.0)는 크로스 런타임 메모리(2.0)의 부분집합.

## 2. 문제 · 포지셔닝 — 빈틈의 재정의 (v0.2)

**디자인 센터 = 에이전트 함대를 지휘하는 사람(Manager of Managers).** 사람 1명이 에이전트 N개를 굴리면 기억의 읽기/쓰기 주체가 1 → N으로 폭증한다. 비동기·트리거 구동 에이전트(노트북을 닫아도 클라우드에서 작업, 이슈/슬랙으로 스스로 기동)의 세계에선 컨텍스트를 공급해줄 사람이 루프 밖에 있다 — **사람이 함대의 '메모리 버스' 역할을 계속할 수 없다**는 것이 구조적 페인이다. (단 이 함대 공유 수요는 검증리포트 6개 need 중 최약체 — N6 '부분 확인', 유기적 요구 얇음, '선택적 공유'가 정답이라는 반증 존재. 연역으로는 강하나 출시로만 실증되는 가설이라, 설계 센터로 두되 초기 검증·출시는 '함대 크기 1'부터 태운다.)

빈틈 프레임 갱신 (2026-07 경쟁 리서치 → `docs/리서치/MCO_경쟁지형_v0_1.md`):

> (구) "빅테크가 하기엔 짜치고, 사용자는 절실한" → **(신) "벤더는 담장 안에서 전부 구현했다(= 수요의 증거). 구조적으로 못 하는 것은 벤더 횡단 중립이다."**

빈틈 3요소의 2026-07 판정:

- **크로스 플랫폼 이동성** — 단독 차별점으로는 사망(Supermemory MCP·mem0 OpenMemory·MemoryPlugin 등 혼잡 시장). 조합의 한 요소로만 유효.
- **계정 바인딩·사용자 소유** — 빅테크 레벨에선 여전히 공백. 저장소 통제권이 전부 벤더/테넌트에 있다.
- **에이전트별 기억 권한 스코프** ★ — 벤더 *안*에서는 각자 구현(수요 검증됨), **벤더 횡단으로는 완전 공백**. nice-to-have에서 필수 인프라로 승격(정밀도 + 최소 권한 보안).

경쟁력의 실체는 단일 기능이 아니라 **현 시점 조합 차별점**이다: **사용자 소유 + 벤더 횡단 권한 스코프 + 정밀 선택 + 게이트 환류 + 브리핑 팩** — 이 조합 전부를 하는 곳은 없음(2026-07 확인). 단 '조합' 자체는 방어 가능한 해자가 아니다 — 사업성 §3 실증대로 아키텍처는 몇 주면 복제되고 최약 조합(mem0)이 채택 1위였다. **방어력의 실제 출처는 조합이 아니라 라벨 복리·유통·기본값 지위**이며, 조합은 그 위에 서는 현재의 차별점으로 서술한다.

**경쟁력의 위치 (사업성 판정, 2026-07-11):** 저장 스택(md+벡터+관계형+스코프 태그)은 **커모디티**다 — 전 요소를 복수 업체가 보유하고 1인 개발자가 무료로 복제한다. 경쟁력은 저장이 아니라 **게이트·정밀 선택·강제 스코프·라벨 축적**에 있다(베이스라인 +10점을 넘는 선택 정밀도는 아직 아무도 못 함). 근거: `docs/리서치/MCO_사업성분석_v0_1.md`.

**반론 방어 — "롱컨텍스트·프롬프트 캐싱이면 메모리 레이어가 불필요한가?"** 아니다: (a) context rot 실증(방해 청크 1개도 정확도 저하, 집중 ~300tok ≫ 전체 113k), (b) 함대 스케일(N런의 읽기/쓰기 주체를 사람이 대신 못 함), (c) 벤더 횡단(한 벤더의 캐시·컨텍스트는 타 벤더로 이어지지 않음), (d) 정밀도=안전(잘못된 주입이 병렬 N런을 오염). 코어는 컨텍스트를 '많이 넣기'가 아니라 '맞게 고르기'다.

개념 앵커 = **Personal Memory Passport** (유지; 단 외부 명칭 충돌 확인 — §9 용어 재검토 참조, 대외 헤드라인 사용 금지).

## 3. 코어 — 스몰 메모리 모델

MCO 안의 **작은 모델**이 메모리 지능을 담당한다. 단순 메모리 RAG가 아니다 — **개인화 히스토리 관리 + RAG + 약간의 자가 학습**이 결합된 형태다. 5역할은 불변(척추)이며, v0.2에서 각 역할의 '함대 시대 직무'가 추가된다:

| # | 역할 | 내용 | 함대 시대 직무 (v0.2) |
|---|---|---|---|
| ① | 기억 가치 판단 | 입력 중 지속 보관할 가치가 있는 정보 식별 | 에이전트 런 배기가스(로그·결과물) 트리아지 — write-back 게이트 |
| ② | 저장 범위 결정 | 어느 스코프(계정·프로젝트·에이전트)에 저장할지 결정 | 법인·프로젝트 벽 배정(멀티 프로젝트 운영자) |
| ③ | 충돌 감지 | 기존 기억과 모순되는 새 정보 탐지 | 에이전트 간 충돌(A는 X, B는 ¬X 전제) 탐지 → 사람의 결정 큐로 |
| ④ | 갱신 | 낡은 기억을 새 사실로 교체·무효화 | TTL·출처 기반 무효화 — 낡은 기억 1개의 병렬 N런 오염 방지 |
| ⑤ | **기억 선택** ★ | 지금 컨텍스트에 필요한 기억만 인출 — 코어 | 위임 시 **브리핑 팩** 생성(매니저→워커 스코프 컨텍스트) |

## 4. 메모리 구조 (구상)

- **수집** — 입력 데이터 + **에이전트 런 배기가스** 중 지속적으로 쓸 만한 기억을 리스트업한다.
- **구조화 (2축)** — 기억 원자 = DB 한 행. 두 축으로 관리한다: **내용 종류(type) 6종**(사실·선호·결정[+근거+기각안]·실패/안티패턴·약속·에피소드요약) × **인출 2계층**(설정성 = 상시 주입 / 사건성 = 검색). *v0.1의 '3층(명시적 저장·맥락 추출·에피소드)'은 도출-출처 관점의 초기 씨앗으로, 이 두 축에 흡수됐다.* 스키마 정본 = `Product/MCO_아키텍처_v0_1.md` §4.
- **선택** — 질문/위임이 들어오면 **필요한 기억만** 꺼낸다. 무관한 기억을 걸러내는 **노이즈 필터**가 코어 가치다(실패 기준: 애플워치 논의에 Setto 기억 주입). 평가 축 = recall보다 **precision 우선**. 함대에선 잘못된 주입이 병렬 N런을 오염시키므로 정밀도는 편의가 아니라 신뢰성/안전 요구사항이다.
- **필수 필드 (코어)** — 모든 기억에 **출처(provenance: 누가·언제·무슨 근거로)·유효기간(TTL·bi-temporal 무효화)·trust_level(신뢰 티어: 사람 단언 > 검증된 런 산출 > 비신뢰 유래 — 보안 통제 ①)**. 저신뢰 기억은 고권한 문맥 자동 주입 금지. 참조 부품: bi-temporal(Zep Graphiti 류).

## 5. 경계 (범위)

- **런(run)/세션 안의 단기 기억은 하네스·LLM 몫이다**(server-side compaction 등). MCO는 **런 밖 — 런 간·에이전트 간·벤더 간 — 지속성 메모리** 전담.
- MCO는 LLM 추론 자체를 하지 않는다 — 컨텍스트를 준비해 **제공**하는 레이어다.
- **가드레일 (v0.2):**
  - MCO는 에이전트를 스케줄링·라우팅하지 않는다 — **오케스트레이션 프레임워크가 아니다**(그건 하네스 몫). MCO가 오케스트레이션하는 것은 '기억'이다.
  - **관측(observability)/트레이싱 제품이 아니다** — 배기가스는 입력일 뿐, 제품은 기억이다.

## 6. 제공 형태

- **MCP 서버** — 어떤 LLM/에이전트든 MCP 연동으로 MCO 메모리를 사용한다. 코어 서버 하나 + 표면별 얇은 어댑터.
- **1차 표면 = Claude 커넥터(웹·데스크톱·모바일) + 웹 대시보드** (v0.3 재개정) — 획득·일반용. 소거법 근거: Claude 커넥터가 **무료 플랜 포함** 유일하게 열린 문(ChatGPT 커스텀 MCP = dev mode 전용, Gemini = 서드파티 주입 경로 없음), 무료는 커스텀 커넥터 **1개 슬롯 = "그 슬롯을 이기는 게임"**(첫 5분이 생존 장치).
- **심화 표면(병행) = 하네스(Claude Code·Cursor) + Hermes 프로바이더** — 자동 주입을 제어할 수 있는 파워용. *(v0.2 '하네스 1차 표면'은 기각이 아니라 병행 구조로 개정 — 하드코어 로컬 설치 UX로는 일반 사용자 획득 불가.)*
- **OAuth 계정 인증** — 기억 = 사용자 계정 소유. 소스 인증으로 서비스한다.
- **권한 스코프** — 계정·법인·프로젝트·에이전트 단위로 컨텍스트를 분리 제공한다.

## 7. MVP 구상 (v0.3 웨지·페르소나)

- 구성: ① **Claude 커넥터 + 웹 대시보드**(1차 표면) ② **MCP 메모리 서버**(코어) ③ **하네스(Claude Code·Cursor)·Hermes**(병행 심화 표면) ④ **결정 인박스형 대시보드** — 충돌 해소·기억 승격/만료·출처 감사만 사람에게 올리는 예외 처리함(CRUD UI 아님). 브라우저 확장(채팅 수집)은 후순위.
- **초기 페르소나 = 컨텍스트가 돈인 프로슈머** — 여러 법인·클라이언트·프로젝트를 병행하며 AI를 채팅 하나가 아니라 여러 도구·에이전트로 굴리는 사람(빌러블 아워·마감·서류·멀티 법인). 디자인 센터 = 에이전트 함대를 지휘하는 사람; 채팅만 쓰는 사용자는 '함대 크기 1'의 특수해(단 함대 공유 수요 = 6 need 중 최약 N6 '부분 확인', 검증된 형태는 무차별 공유가 아니라 '선택적 공유' — 초기 출시는 '함대 크기 1'부터). **지불 실측은 아웃컴(경로 B 서류 컴포저)에만 있고 코어 메모리 레이어 WTP는 미검증(매출 공개 메모리 회사 0곳)이라, 코어 지불 의사는 인터뷰가 아니라 실출시·실결제로 검증한다.** *'코딩 에이전트 파워유저'는 최상위 페르소나가 아니라 **경로 A의 초기 획득 채널**(MCP 네이티브·페인 즉시성·전파력).*
- **GTM 경로 A·B 병렬 실험** (사업성 판정): **A** = 슬롯 정밀도 승부(코딩+커넥터), **B** = 아웃컴 앱(서류 컴포저 — 지불 실측 유일, 수동 큐레이션 시작 가능). 둘 다 ~공짜 실험, 빌드 순서는 결과로. (C 엔터프라이즈 = 지금 아님.)
- **최우선 기술 검증: write-back 정밀도** — 실제 에이전트 세션 로그에 역할 ① 트리아지를 돌려 건진 기억의 precision을 수동 채점(코드 이전에 가능). 이게 안 되면 원장이 노이즈로 차서 2.0 전체가 무너진다.
- **웹 표면의 정직한 한계** — 서드파티 기억은 모델이 도구를 호출해야 작동(자동 주입 아님) → 첫 경험은 '저절로 아는'이 아니라 **'연결하자마자 보여주는'**(웜스타트 스캔·부패도 검진 = 우리가 제어하는 호출로 100% 재현). 마케팅 금지어: "GPT에서도 딸깍"(현재 지킬 수 없는 약속).

## 8. 버티컬 — 제조

(v0.1 유지) 제조업 적용 시 **'LLM 독립형 메모리 레이어' 표현을 쓰지 않는다.** 정본 포지셔닝:

> **"현장의 설비·품질·작업자 이력을 기억하고, 지금 상황에 딱 맞는 과거 사례를 꺼내주는 제조 메모리 컨텍스트"** (Factory Memory Context)

'지금 상황에 맞는 과거 사례 인출' 라인으로 시작해야 설득력이 높다(**가설 — 제조 사용자·페인은 미검증, §9 [미결]. 현장 인터뷰 1~2건으로 페인 존재부터 확인한 뒤 정본화**).

## 9. 미결 (백로그)

논의로 확정할 것들. 상태 = **[미결] / [부분확정→정본] / [방향만]**. 확정 시 본 차터 갱신 + `.claude/memory/` 기록:

- **[부분확정→정본] 메모리 스키마** — 기억 원자 스키마(type 6종·2계층·provenance·TTL·trust_level)는 `Product/MCO_아키텍처_v0_1.md` §4로 확정. 잔여 = 충돌 해소 규칙 세부·스코프 모델 형식화.
- **[부분확정→정본] 프라이버시·보안 모델** — 아키텍처 제약 본체(통제 6항: 신뢰 티어링·write-back 스코프 고정·스캔=보조·대시보드 IAP·신뢰 헤더·제로 상태 테스트) 확정, 정본 = [[mco-security-model]] + 아키텍처 §6. 잔여 = 암호화 수준·삭제권 보증·보존 정책.
- **[방향만] 수익 모델** — 획득 = 무료 + 출력 순간 / 전환 = 무료가 못 주는 것(횡단 동기화·벽·팀·정밀). **GB 저장 가치 미터 기각**(매출∝저장량 = 축적 최적화 → 노이즈 → 품질 논거 자충). Pricing은 상품성(needs·UX·기술) 결론 전까지 유보([[mco-thinking-order]]).
- **[미결] 용어·서비스 네이밍 ⚠️** — 'Personal Memory Passport' 외부 충돌(MemoryLake 제품명·mem0 비전 문구). 상표 조사 + 앵커 유지/교체(대외 헤드라인 사용 금지).
- **[미결] 오픈 메모리 포맷 표준 전략** — '에이전트 메모리' 오픈 표준 부재(AGENTS.md·MCP는 AAIF 이관). 선제 제안 플레이 검토(AAIF/W3C CG 창구).
- **[미결] 스몰 모델 실체** — 소유(파인튜닝·호스팅·온디바이스) vs 렌탈. 자산 본체 = 가중치가 아니라 **스키마 + precision 평가셋** 가설.
- **[미결] 수집 경로** — 하네스·MCP·웜스타트 외 추가 소스.
- **[미결] 제조 버티컬 구체 시나리오.**

## 10. 용어

| 용어 | 정의 |
|---|---|
| **MCO** | Memory Context Orchestrator — 본 프로젝트 정본 명칭 |
| **Personal Memory Passport** | 크로스 플랫폼 메모리 이동성의 개념 앵커 (⚠️ 외부 충돌 — §9 재검토, 대외 헤드라인 사용 금지) |
| **스몰 메모리 모델** | MCO 내부에서 5역할을 수행하는 작은 모델 |
| **기억 선택** | 상황에 필요한 기억만 인출하는 코어 기능(역할 ⑤) |
| **브리핑 팩** (v0.2) | 위임이 일어나면 MCO가 **공급하는** 스코프 컨텍스트 묶음(위임 판단·라우팅은 하네스·매니저 몫 — MCO는 오케스트레이션 안 함) — 기억 선택의 위임 버전 |
| **환류 / write-back** (v0.2) | 에이전트 런의 학습이 가치 판단 게이트를 거쳐 지속 메모리로 유입되는 흐름 |
| **결정 인박스** (v0.2) | 충돌 해소·승격/만료·출처 감사를 사람에게 올리는 아우터 루프 표면 |
| **trust_level** (v0.3) | 기억의 신뢰 티어(사람 단언 > 검증된 런 산출 > 비신뢰 유래) — 저신뢰는 고권한 문맥 자동 주입 금지(보안 통제 ①) |
| **경로 A·B** (v0.3) | GTM 병렬 실험 — A=슬롯 정밀도(코딩+커넥터), B=아웃컴 앱(서류 컴포저) |
| **제조 메모리 컨텍스트** | 제조 버티컬 정본 포지셔닝 (Factory Memory Context) |
