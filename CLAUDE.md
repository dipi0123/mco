# MCO — 작업 지침 (CLAUDE.md)

> 작업 폴더가 열릴 때 자동 로드되는 **진입점**이다. 프로젝트 정체성 + 메모리 운영 규칙을 고정한다. 도메인 규정의 단일 진실원천은 **`docs/charter.md`** (Project Charter).

## 1. 프로젝트

**MCO (Memory Context Orchestrator)** — 사용자의 메모리를 관리하고, 프롬프트를 해석해 **계정·프로젝트·에이전트 단위**로 필요한 컨텍스트만 추출해 LLM/에이전트에 제공하는 **메모리 오케스트레이션 레이어**. LLM 대체가 아니라 그 **앞단 계층**('메모리 OS')이다.

코어/차별점:

- **크로스 플랫폼 메모리** — 각 LLM의 자체 메모리는 플랫폼 밖으로 이어지지 않는다. MCO는 **MCP 서버 + OAuth 계정 바인딩**으로 사용자 소유 메모리를 GPT·Claude·Cursor·커스텀 에이전트 어디서든 일관 제공. 개념 앵커 = **Personal Memory Passport**.
- **스몰 메모리 모델** — MCO 안의 작은 모델이 5역할: **기억 가치 판단 · 저장 범위 결정 · 충돌 감지 · 갱신 · 기억 선택**. 단순 메모리 RAG 아님 — 개인화 히스토리 관리 + RAG + 약간의 자가 학습.
- **저장 < 선택** — 상황에 필요한 기억만 인출하고 무관한 기억을 걸러내는 **노이즈 필터**가 코어 가치.
- **경계** — 세션(단기) 기억 = 기존 LLM 몫. MCO = **세션 밖 지속성 메모리** 전담.
- **포지셔닝** — 빅테크 빈틈("빅테크가 하기엔 짜치고, 사용자는 절실한") 공략. 제조 GTM 정본 = **제조 메모리 컨텍스트**(Factory Memory Context).

## 2. 메모리 운영 ★

- 지속 메모리는 **이 repo의 `.claude/memory/`** 에 둔다. 앱 내부 메모리 영역(Application Support)은 기기 로컬이라 git 공유가 안 되므로 **쓰지 않는다.**
- 세션 시작 시 **`.claude/memory/MEMORY.md` 인덱스를 읽고** 관련 메모리 파일을 로드한다.
- 새 지속 사실은 `.claude/memory/` 에 파일로 기록하고 `MEMORY.md` 인덱스에 한 줄 추가한다. 한 파일 = 한 사실, frontmatter(`name`/`description`/`metadata.type`) 포함, 관련 메모리는 `[[name]]` 로 링크.
- **브레인스톰 캡처** — 세션에서 새 합의·결정·기각이 나오면 그 자리에서 메모리에 기록한다(기각도 기록 — 같은 논의 반복 방지).

## 3. 컨셉 코어 — 현재 합의 ★

브레인스토밍 단계다. 잠긴 설계는 아직 없다 — 모든 결정은 **현재 합의**이며 merit로 재론 가능. 단 정본 용어·코어 표현은 임의 변경 금지(바꾸려면 채팅 정렬 먼저):

- 명칭 정본: **MCO = Memory Context Orchestrator**.
- **스몰 모델 5역할**(가치 판단·범위 결정·충돌 감지·갱신·선택)과 **저장<선택** 원칙 = 컨셉의 척추 — 모든 산출물에서 전면 유지.
- 제조 맥락에서 **'LLM 독립형 메모리 레이어' 표현 금지** → **'제조 메모리 컨텍스트'** 로 포지셔닝.
- 상세: `.claude/memory/mco-concept-core.md` 및 인덱스 참조.

**워크플로:** 큰 결정은 채팅에서 정렬한 뒤 Ethan이 **"작성"** 으로 생성을 트리거한다. 그 게이트를 존중하고 임의 선행 생성 금지.
